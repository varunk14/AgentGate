# DECISIONS.md — AgentGate design decisions and their reasoning
This file records the design judgment behind AgentGate. It is a first-class artifact: a reader should finish it thinking "this person knows exactly what they built, and where the value stops." Each entry states the **decision**, the **reasoning**, and the **honest boundary/limitation**. Decisions come from a deep design review; the product spec defines what to build, this file is the *why*.

---

## 0. Threat model (read this before anything else)

> **AgentGate is an agent-reliability gate, not a security/trust gate.** v1 catches an **honest-but-fallible agent's mistakes** — arithmetic slips, decimal errors, LLM misreads, duplicates — at the moment of the write, and returns a machine-readable reason the agent can fix against. It does **NOT** defend against an **adversarial agent that forges its own evidence**, and it does **NOT** detect fraud or bad business judgment. Passing AgentGate means "the action is consistent with the evidence provided," never "the payment is correct or authorized."

**Reasoning.** Every AgentGate answer leans on the word *source*. If the untrusted agent supplies both the proposed action and the evidence, and we trust that evidence, the gate is theater. Being explicit about this is what makes the rest of the repo credible to a threat-modeling reader — the same fact framed honestly reads as "knows what they built," not "caught overselling."

**Boundary.** v1's honesty is precisely this scoping. The adversarial case was a named milestone (D5) and has since shipped as fetch mode (D45) — the upgrade holds for deployments that configure an operator-controlled system of record; caller-supplied mode retains this boundary in full. Fraud and business-judgment are permanently out of scope — that is what ESCALATE plus a retained human reviewer are for.

---

## D1 — Money is `Decimal`, never float
**Decision.** All money is Python `Decimal` parsed from the source **as a string** (or integer cents). Never float. The `$0.01` arithmetic tolerance is reserved for real vendor rounding, never for masking float error.
**Reasoning.** `0.1 + 0.2 != 0.3` in floats. A "trustworthy deterministic core" built on floats would generate its own rounding lies and quietly break the exact trust the product claims. This is the one place a sloppy type choice undermines everything, so it is `Decimal` from the first line. `Decimal("1240") == Decimal("1240.00")` is `True`, which also makes grounding (D10) clean.
**Boundary.** Requires disciplined parsing (string → Decimal) at every ingestion point; a single `float()` in the path reintroduces the bug.

## D2/D16 — Scoring: no self-reported confidence; confidence = grounding coverage
**Decision.** `score = (soft-check pass ratio) × (fields grounded / fields total)`. For a structured source with no LLM in the path, grounding = 1.0. Critical checks are **not** in the score — they are a separate hard gate. **No self-reported LLM confidence is used anywhere.**
**Reasoning.** An LLM emitting "I'm 0.97 confident" is uncalibrated theater — it will say that while hallucinating, so letting it multiply a trust score is a fabricated number. The only honestly *earned* confidence is grounding coverage: the fraction of extracted fields that literally verify against the source. So confidence is not a separate input — it **is** the grounding result. Every factor in the score is measured against something external; nothing is self-reported. This also makes the `score_below` policy knob (D3/§8) a live, meaningful control rather than a dead one.
**Boundary.** The score reflects *how much was grounded*, not *whether the invoice is truthful*. Full grounding coverage on a doctored source still yields a high score (see D5).

## D3 — Decision precedence: BLOCK > ESCALATE > ALLOW
**Decision.** An **agent-fixable** failure **short-circuits to BLOCK** before escalation is evaluated. A critical failure that is *not* agent-fixable does **not** BLOCK — it ESCALATES. ESCALATE applies to non-agent-fixable critical failures (source-invalid arithmetic, currency mismatch, adjustment difference) and to soft-check escalations (vendor ambiguity, duplicate, low score, fail-closed). ALLOW only when every critical check passes and nothing escalates.
**Reasoning.** BLOCK is provably-wrong-and-machine-fixable — cheap, instant, and it feeds the agent's retry loop. ESCALATE is plausibly-right-but-risky — expensive human time. The earlier "any critical failure → BLOCK" was too coarse: a broken *source* or an unverifiable currency/adjustment is not something the agent can correct by re-rolling a number, so BLOCKing it would spin the retry loop pointlessly. BLOCK is therefore reserved for the one case the agent can actually fix (the amount misread, D13); every other critical failure is human work → ESCALATE (fail-closed). See D7 for `agent_fixable` vs `source_invalid` block types.
**Boundary.** Correct routing depends on correctly classifying each failure as agent-fixable vs not; a miscategorized failure sends work to the wrong lane.

## D4/D10 — Grounding semantics: literal presence, matched on parsed Decimal
**Decision.** Grounding checks that the claimed number **literally appears in the caller-supplied `raw_text`**, matched on the **parsed Decimal value** (not the raw string), at **token level, not substring** (see D21): tokenize money-shaped values, parse each to Decimal, compare Decimals. It catches extractor hallucination and honest mistakes — not source integrity.
**Reasoning.** The LLM may emit `1240`, `1,240.00`, `$1,240.00`, or `1240.0` while the source says `1,240.00`. All parse to exactly one Decimal with no information lost — a lossless canonical form (see D8's principle). So matching on Decimal value is still fully deterministic. Grounding defends against *our own extractor*, confirming the number is present in the document rather than invented.
**Boundary.** Because `raw_text` is caller-supplied (D5), grounding proves the number matches the text the agent handed us — not that the text is genuine. A hostile caller submitting doctored `raw_text` plus a matching action passes grounding.

## D5 — Trust anchor: v1 caller-supplied; independent fetch is the milestone
**Decision.** v1 accepts caller-supplied `source` (invoice, raw_text, po) via the API — it verifies **consistency of what it is given** and does **not** independently fetch the source. The named upgrade: **independent source fetch from a system of record (ERP/accounting/ledger), keyed by an identifier the agent provides** — the agent chooses *which* invoice, never *what it says*.
**Reasoning.** The honest scoping of the whole product. If AgentGate trusted an agent-supplied document body it would be pointless against an adversary; being explicit converts an overclaim into a credible, staged roadmap. The milestone is what upgrades "consistent with what it's given" → "checked against a source the agent can't control."
**Boundary.** v1 is defenseless against forged evidence, by construction. This is stated at the top of the README and this file — a deliberate boundary, not a hidden gap.
**Postscript (shipped).** The milestone shipped as **fetch mode** (D45/D46): `source: {"fetch": "<invoice-number>"}` resolves the record from an operator-configured system of record, with `system_of_record:`-prefixed evidence on the wire. The upgrade is deployment-scoped — it holds where the records store is writable only by the operator; the v1 reference store is a local directory, and a live ERP connector remains open. Caller-supplied mode remains available and retains exactly the boundary stated above.

## D6 — Eval built to embarrass, reported honestly
**Decision.** The metric is **not** "block-accuracy." Report a confusion matrix as **precision, recall, and false-positive rate separately**. The dataset must include (a) **false-positive traps** — legit invoices with valid-but-unusual features (rounding lines, alternate vendor spelling, multi-currency, credit-note-like) that a naive check would wrongly BLOCK; (b) **out-of-scope cases** — consistent-but-actually-wrong invoices (double-bills, unauthorized spend) labeled as known misses; (c) honest-agent errors it does catch.
**Reasoning.** A dataset the author writes to match their own checks trivially reports ~100% and is theater. A false-BLOCK on a legit invoice is the exact failure that destroys trust in production, so it must be measured. Reporting real numbers below a clean 0.9, plus near-zero recall on out-of-scope cases *by design*, is more credible than a rigged 100%.
**Boundary.** The honest numbers will look worse than a rigged benchmark; that trade is intentional.

## D7 — Block payload is compiler-grade and typed
**Decision.** A BLOCK returns a structured, machine-readable reason: `check`, `expected`, `received`, `field_to_change`, `message`. Two block types: **`agent_fixable`** (action misread a valid source → agent may retry, capped at 2) and **`source_invalid`** (the source itself is internally inconsistent → do NOT retry, route straight to ESCALATE). The retry loop terminates on fix, on cap, or immediately on a source-invalid block.
**Reasoning.** "Structured reason" is only useful if it is mechanically actionable — like a compiler, it must say *what* and *where* ("set `proposed_action.amount` to 1240.00"), not just "wrong." Otherwise the agent re-rolls the dice and the compiler-error analogy is dead. And some blocks are unfixable by the agent because the *source* is wrong, not the action; resubmitting forever would never converge, so those must escalate immediately.
**Boundary.** The retry cap (2) is a deliberate ceiling; a genuinely fixable action that needs 3 tries will escalate. Distinguishing agent-fixable from source-invalid requires the arithmetic checks (D15) to attribute the inconsistency correctly.

## D8 — Vendor matching: cosmetic-normalize (deterministic) vs entity-diff (escalate)
**Decision.** Cosmetic normalization only — case, whitespace, trailing punctuation — is deterministic and allowed (exact match after it). Entity-level differences ("Acme Corp" vs "Acme Corp Holdings" vs "Acme Corp LLC") are **never** auto-normalized. Anything not an exact match after cosmetic normalization is a **soft failure that ESCALATES**, never an automatic allow or block.
**Reasoning.** Too strict → false-BLOCK a legit vendor (the trust-killing false positive). Too loose → ALLOW payment to a different legal entity (a real diversion vector). So we refuse to be either: cosmetic exact-match stays in the deterministic core; fuzzy entity matching lives explicitly as an escalation trigger, not a hidden judgment smuggled inside a "deterministic" check.
**Boundary.** Legitimate vendor-name variations (a genuine rename, a d/b/a) will escalate to a human rather than auto-pass. That is the intended cost of not guessing entity identity.

**Principle (D8 + D10).** *Canonicalize only when the transform is lossless and unambiguous* (all number formats → one Decimal). *Escalate when it would require discarding meaningful information* (entity identity). Number-for-grounding is the former; vendor-fuzzy is the latter — same rule, correctly applied, not a double standard.

## D9 — Test seam: substitution vs fault injection; the router is the only substitution
**Decision.** **No verification logic is ever mocked when its verdict is asserted. The LLM router is the only substitution seam.** Tests inject stubbed raw LLM outputs (including messy/malformed) through `llm_router.py`'s interface and run the real parsing/grounding/decision logic **unmocked**. Distinct and allowed: **fault-injection doubles** for failure paths that cannot otherwise be reached — a `decide` patched to *raise* (exercising the boundary catch-all, D35) and a raising tracer (exercising observer isolation, D37). A fault-injection double never returns a canned verdict; every test that asserts decision *content* runs the real logic. A `live`-marked real-call test targets the agent's proposal step (the one model call in the system); run manually, never in CI.
**Reasoning.** The two kinds of double have opposite intents, so "only mock" needs the distinction to stay honest rather than literal-but-false. A **substitution** mock replaces real logic with canned output, so bugs hide behind it — that is what this rule forbids everywhere except the network call itself. A **fault-injection** double forces a failure that *adds* coverage: the catch-all contract and the tracer-isolation guarantee are untestable without one, and nothing hides behind it because no verdict content is asserted through it. CI stays fast, free, and deterministic, and can't be broken by a provider rate-limit mid-run.
**Boundary.** CI never exercises the real provider, so provider-contract drift (a model changing its output shape) is only caught when the live test is run manually — and an unrun live test is coverage on paper only; the requirement is satisfied by its first green run, not by the marker existing.

## D11 — Fail-closed + provider abstraction
**Decision.** The router forces every provider through a lowest-common-denominator contract: **text in → JSON out, validated by us with Pydantic**, never trusting native JSON mode or function-calling; provider-specific glue is centralized in the router (that is what makes the "one-line swap to paid" claim honest). On any failure — malformed JSON, rate-limit, timeout, extraction failure — AgentGate **never crashes and never ALLOWs**: it surfaces a **typed error** that the decision layer converts to **ESCALATE**. *If it can't verify, it doesn't approve.*
**Reasoning.** A "verification gate" that crashes or silently allows when its own LLM hiccups isn't a gate. Failing safe-to-human is a core behavior, not an afterthought. Centralizing glue keeps business logic model-agnostic and the swap genuinely one line elsewhere.
**Boundary.** The LCD contract means we deliberately forgo provider-specific niceties (native structured output) in favor of our own validation. Frequent LLM failures degrade gracefully to more human escalations, not to throughput.

## D12 — Currency is first-class and critical
**Decision.** Money is `{value, currency}` on both invoice amounts and the proposed action. `action.currency == invoice.currency` is a **critical** check, exact, with no conversion. Currency mismatch, or **dual-amount ambiguity** (an invoice printing both a foreign and a converted local amount), → **ESCALATE**. v1 does not perform or verify FX conversion.
**Reasoning.** Without currency in the checks, AgentGate would wrongly ALLOW paying 1,240 USD against a 1,240 EUR invoice — a real bug. We don't guess which of two printed amounts is "the total"; ambiguity means "can't verify," which fails closed to escalate (D11), never allow.
**Boundary.** Any genuinely multi-currency or FX-converted invoice escalates in v1; automated FX verification is out of scope.

## D13/D17 — Amount-vs-total: exact ALLOWs; declared-adjustment difference ESCALATES; undeclared difference BLOCKS
**Decision.** `action.amount == invoice.total` exactly → eligible for ALLOW. A difference is split by whether the agent **declared an adjustment**:
- Difference **with** a non-empty `adjustments` (withholding, discount, installment) → **ESCALATE** — may be legitimate but is unverifiable in v1.
- Difference **with no** declared adjustment → **BLOCK** (`agent_fixable`): the action misread a valid total; return `field_to_change: proposed_action.amount` so the agent self-corrects.
Adjustment verification (grounding each adjustment) remains a **named later milestone**, not v1.
**Reasoning.** Two truths were in tension. (1) `action.amount == invoice.total` is wrong for real finance — a $1,000 invoice with 10% withholding correctly remits $900 — so blindly blocking every difference is the trust-killing false positive. (2) The product's hero feature is the compiler-error retry loop, whose canonical case is a decimal slip (propose 12,400 against a 1,240 total); if *every* difference escalated, that case would burn a human and the retry loop would never fire. The declared-adjustment flag is the deterministic signal that separates them: no declared intent to pay something other than the total ⇒ treat the mismatch as a misread the agent can fix (BLOCK); a declared adjustment ⇒ a possibly-legitimate payment we cannot verify (ESCALATE, fail-closed). This supersedes the earlier "any difference → ESCALATE" stance, which would have killed the hero feature.
**Boundary.** A real adjusted payment that the agent forgot to declare will BLOCK, not escalate — the agent must declare the adjustment (then it escalates for human review). Declared adjustments are never auto-verified in v1; that is the deferred milestone.

## D14/D18 — Credit notes out of scope for v1
**Decision.** Credit notes (negative totals) are **out of scope for v1**, documented as a limitation. Either grow `action_type` later or keep them out.
**Reasoning.** Decided by thesis-value, not completeness. If a credit note is "approve a negative amount," it exercises the same machinery and proves nothing new about "does the action agree with the evidence." If it needs different logic (matching a credit to an original invoice), that's domain scope creep that still adds no thesis proof. Forcing them through `approve_payment` would be a modeling error.
**Boundary.** No credit-note handling ships in v1; it lives in documented limitations.

## D15 — Structure-aware arithmetic (survives real invoices)
**Decision.** Replace naive `sum(line_items) == subtotal` and `subtotal + tax == total` with a structure-aware check: **`(Σ charge lines incl. shipping − Σ typed discount lines + Σ tax summed across multiple rates) == total`** (within `$0.01` for real vendor rounding). Line items carry a `kind` (charge | discount | shipping | tax). Exotic formats (complex multi-page, unusual regional tax, bundled/prorated) → ESCALATE, documented.
**Reasoning.** A normal real invoice has a discount line, shipping, and two VAT rates — which false-BLOCK the naive checks, making the tool useless in production. Widening the check to reflect real invoice structure keeps it fully deterministic *and* true on real inputs. The principle throughout: widen the check enough to be true on real inputs, escalate what you can't verify, never fake coverage or force-check.
**Boundary.** Invoices whose structure the check doesn't model escalate rather than being force-checked; correctly attributing a mismatch to `source_invalid` (D7) depends on this check. A total-only invoice (no line items and no tax lines) is *unverifiable*, not inconsistent — it escalates with a "nothing to verify" reason (no `block_type`), never `source_invalid`: calling it source-invalid would tell a human the vendor's document is broken when the truth is there was nothing to check it against.

## D19 — Slice 1 emits a grounding result, not a decision
**Decision.** The Slice 1 skeleton does **not** emit `allow | block | escalate`. Its output is a grounding result: **`grounded | not_grounded | ungroundable`**. It means "the number appears in the source," never "the payment is correct." The real decision arrives in Slice 2 once the deterministic checks exist.
**Reasoning.** In Slice 1 there is grounding but no arithmetic checks, so a green ALLOW there could be a payment Slice 2 would BLOCK — it would falsely signal "verified" when only "grounded" is true. Labeling the output as grounding-only keeps the walking skeleton honest: it proves the extract→ground→decide pipeline runs end to end without masquerading as a trustworthy approval.
**Boundary.** Slice 1 makes no correctness claim about the payment; it is a disposable skeleton whose job is to de-risk the hard path (extraction + grounding on a messy source), not to approve anything.
**Postscript (post-v1 cleanup).** "Disposable" was eventually taken at its word: the skeleton pipeline (`assess_grounding`/`extract_total`) was deleted once the full system shipped. It had no production caller — the serving paths deliberately run no LLM (D35/D44), and its output (a total + verdict) could not feed `decide()`, which needs a full structured Invoice — so it was dead code, not a library entry point. What survives from Slice 1: the deterministic token-level grounding matcher (D21; now the coverage primitive, D27) and the messy-JSON recovery (`parse_llm_json`, module `llm_json.py`), consumed by the demo agent's proposal step (D30) — which is now the only model call in the system. The verification path is fully deterministic: regex and Decimal only.

## D20 — North-star: general gate for all agents (DEFERRED to post-v1)
**Decision.** AgentGate's true intent is a **general pre-action verification gate for any agent**, not an invoice tool — the name is *Agent*Gate, and invoices are only the demo domain. The target architecture is a **domain-agnostic engine** (grounding, decision + precedence, scoring, fail-closed, BlockReason + retry contract, LLM router, REST/MCP interface — none of which know about invoices) plus **pluggable domain "packs"** (concrete schemas + deterministic checks). Invoices are the first, deep **reference pack**; a thin second pack (e.g. a refund/order bot: amount ≤ order total, currency match, not-already-refunded) later **proves** the seam is real. **This is DEFERRED: v1 builds the invoice vertical first and only generalizes once it works end-to-end.**
**Reasoning.** "Works for all agents" is earned by depth in one real domain plus a visible seam — not by an abstract framework that credibly verifies nothing (the Pydantic-validation objection). The universal primitive is **grounding** ("does a claimed value literally appear in the source evidence?"), which is domain-independent; domain packs add structural checks on top. Building the engine general *in interface* costs almost nothing (mostly: don't import invoice types into engine logic); building it deep up front would be over-engineering and would delay the thesis proof. So we prove invoices first, then add a second pack as the generality proof.
**Boundary.** v1 ships invoice-only in behavior. Generality is a documented direction, not a v1 claim. Coding hygiene during v1: keep grounding/decision/scoring free of invoice-specific imports where free, so the later engine/pack split is a re-shelving, not a rewrite.

## D21 — Grounding is token-level money-value matching, not substring
**Decision.** Grounding tokenizes `raw_text` for **money-shaped values** (optional currency symbol, thousands separators, decimal places) with boundaries, parses each candidate to `Decimal`, and compares Decimals. It does **not** do substring presence. So `INV-31240`, a date `12/40`, or `$11,240.00` never yield a spurious `1240`.
**Reasoning.** A substring scan grounds `1240` inside `INV-31240` and manufactures confidence for a number that was never the total — worse than no grounding, because the score (D2/D16) then multiplies trust by a false signal. Token-level Decimal matching removes that class of false positive while staying fully deterministic (the lossless-canonical principle, D8/D10).
**Boundary.** Even token-level matching proves "this value exists as a monetary value somewhere in the source," **not** "this value is the total" — necessary, not sufficient. That is exactly why the deterministic arithmetic check (D15) sits alongside grounding rather than grounding standing alone. Guard test: `raw_text` with `INV-31240`, `12/40`, `$11,240.00` and no real `1240.00`, claimed total `1240.00`, must return `not_grounded`; a substring matcher fails it immediately.

## D22 — Tests must have teeth (mutation-checked, real messiness)
**Decision.** Test inputs are not hand-picked to pass. Canned router outputs reproduce **real LLM messiness** (markdown-fenced JSON, chatty preamble, trailing commas, string-vs-int values, null fields, truncated JSON); grounding fixtures use realistic invoice text. Every test is written **red-first** (watched to fail for the right reason) and the suite is **mutation-checked**: a plausible wrong implementation (grounding hard-coded to `true`, or Decimal-equality swapped for substring) MUST turn at least one test red.
**Reasoning.** A test where the author controls both sides only proves equality works. A suite in which no plausible wrong implementation reddens anything is decorative — it manufactures confidence exactly like the substring grounding it fails to catch. Red-first plus mutation is what converts "4 passed" from a checkmark into evidence.
**Boundary.** Nothing fully tests the tests; this is why the Slice 4 eval harness exists as an independent signal on data the checks were not designed around. Mutation testing here is applied by judgment, not an automated mutation tool, in v1.

## D23 — "Never weaken a test" made enforceable via the spec
**Decision.** When a test goes red, resolve it against the **spec, not taste**. If the expected value follows from the spec → the code is wrong, fix the code. If the expected value contradicts the spec → the test is wrong, fix the test **and cite the specific spec line in the commit message**. If the spec is silent or ambiguous → stop, resolve with the human, update the spec first, then fix whatever now disagrees. A test edit with **no spec citation is the smell**.
**Reasoning.** "Fix the code" and "fix the test" both end in green, so "never weaken a test" is unenforceable without an external tiebreaker. Making the spec the arbiter, and requiring a citation on any test-expectation change, turns weakening from an invisible judgment call into a reviewable, visible act.
**Boundary.** Relies on the spec being kept current. A stale spec makes the arbiter wrong; that is why spec updates precede behavior changes.

## D24 — Full-suite regression on every commit
**Decision.** The gate for Slice N is "**all tests, Slices 1..N, green**," enforced by CI on every commit — not just the current slice's tests.
**Reasoning.** Running only the current slice's tests means "each slice works" holds only on the day it was built; the next slice can silently invalidate it (a house of cards). A later slice reddening an earlier test is the discipline working — a real regression caught — and D23 then decides whether the code is wrong or the spec changed.
**Boundary.** Affordable precisely because CI is fully stubbed and deterministic (D9): no live LLM calls, no rate limits, so running everything every time stays fast and free. Live-marked real-provider tests remain outside CI.

## D25 — The retry loop never rewrites the gate's verdict
**Decision.** The retry loop consumes a `Decision` and may resubmit a fixed action, but it **never mutates** the `Decision` returned by `decide()`. The loop reports its own outcome separately (a `resolution`: allowed / escalated-by-gate / escalated-to-human). When the loop gives up — cap exhausted or the block is unfixable — the returned gate `Decision` still reads BLOCK; the escalation is expressed as the loop's resolution, not by turning BLOCK into ESCALATE on the Decision.
**Reasoning.** `decide()` is pure and its output is the product's core artifact — the thing that gets logged, traced, and shown in the dashboard. If any caller can rewrite BLOCK into ESCALATE, then a `Decision` in a log means two different things depending on who produced it, and the record becomes untrustworthy. Cap exhaustion must still fail closed (don't retry forever, don't approve), but that safety behavior belongs to the *loop*, reported by the loop, not smuggled into the gate's verdict. This keeps a one-to-one meaning between a `Decision` value and what the checks actually found.
**Boundary.** Consumers must read the loop's `resolution` (not just `final_decision.decision`) to know whether a BLOCK was routed to a human. That is the intended contract: the verdict says what the evidence showed; the resolution says what the loop did about it.

## D26 — Eval semantics: interventions vs. ground truth, and escalations are not free
**Decision.** Every eval case carries two independent labels: **ground truth** (`truth_wrong` — is the proposed action actually wrong?) and the **spec-expected decision** (what the gate must return per the checks). An **intervention** is any non-ALLOW decision (block or escalate). Precision, recall, and false-positive rate are computed on intervention-vs-truth: **escalating a perfectly legitimate payment counts as a false positive**; out-of-scope wrongs the gate ALLOWs by design (renumbered double-bills, forged-but-consistent sources) count as recall misses and are reported by name; **in-scope recall** is reported alongside so "catches what it claims to catch" stays visible. The harness runs the real `decide()` over structured invoices with no LLM in the path — fully deterministic, runs in CI.
**Reasoning.** Fail-closed routing sends legit-but-unverifiable payments (a declared withholding, a vendor rename, a total-only invoice) to a human. That is correct behavior AND a real cost — a reviewer spends attention on a fine payment. Scoring those escalations as successes would let the gate escalate everything and look perfect; scoring them as false alarms makes precision a live measure of the human attention the gate burns. Symmetrically, the threat model (§0) promises specific misses, so the eval must show them dragging overall recall below 1.0 rather than hiding them.
**Boundary.** The metrics are relative to a hand-labeled dataset the author wrote; they demonstrate honest accounting, not field performance. The eval is structured-only and stayed that way by design: grounding coverage was later wired into the score (D27), but the harness supplies no `raw_text`, so coverage is 1.0 there (D27 boundary) — the prediction here that the LLM path would "enter the eval" never came true, and the extraction pipeline itself was later removed as dead code (D19 postscript).

## D27 — Grounding coverage in the score, and the decisive total gate
**Decision.** When `raw_text` is supplied, `decide()` computes grounding coverage over the money fields a deterministic check consumes — `total`, every `line_item.amount`, every `tax_line.amount` — as `fields_grounded / fields_total`, and multiplies it into `score = soft_pass_ratio × coverage` (D2/D16). `subtotal`, `unit_price`, `quantity` are excluded (no check consumes them). No `raw_text` → coverage `1.0`. Independently and decisively: if `raw_text` is supplied and `invoice.total` does not ground, `decide()` ESCALATES (reason `total_not_grounded`, `block_type` none) regardless of coverage; this is **not** folded into `score_below` or the soft-check ratio. `decide()` stays pure/LLM-free — grounding (`is_grounded`) is deterministic regex+Decimal; only extraction uses the router, and that happens before `decide()`.
**Reasoning.** Coverage is a ratio, so a single ungrounded field is diluted by invoice size — an ungrounded `total` on a 20-field invoice scores ~0.95 and clears `score_below: 0.80`, ALLOWing a hallucinated total, the exact failure D4 grounding exists to catch. The total is the one number the decision turns on (action.amount is checked against it), so its grounding must be a hard gate, not an averaged term; sensitivity cannot scale with invoice size. Grounding the fields a check consumes (and only those) keeps the score measuring something the decision depends on rather than padding it with unconsumed values (the same rule that excludes `unit_price`/`subtotal`). Keeping the coverage computation in `decision.py` rather than `grounding.py` preserves that module's domain-agnostic hygiene (D20).
**Boundary.** Coverage inherits grounding's presence-only, currency-blind semantics (D21): two `200.00` line items are both satisfied by a single `200.00` token in `raw_text` (intended — presence, not position), and value match ignores currency (the critical `currency_match` check carries currency). On structured-only inputs (no `raw_text`, e.g. the eval harness) coverage is 1.0, so `score_below` is inert there and only bites on the served `raw_text` path.

## D28 — Policy adds escalations; it never opens the gate
**Decision.** `policy.py` loads `default.yaml` into a typed `Policy`; every numeric threshold is parsed to `Decimal` (via `str()`), never float (D1 extends to policy). `amount_greater_than` and `score_below` add ESCALATE triggers evaluated **only in the non-BLOCK branch**, preserving BLOCK > ESCALATE > ALLOW (D3); a firing threshold appends a `BlockReason` (`block_type` none) to `Decision.reasons` and is not added as a verifier check row. `critical_checks` is an **asserted invariant** — the loader raises `PolicyError` unless the YAML set exactly equals the checks the verifier marks `CheckKind.critical`. There is deliberately **no** `block_if.any_critical_check_failed` flag.
**Reasoning.** A reliability gate whose safety can be disabled by a config typo is not a gate, so config may only *add* escalations, never relax the deterministic core (fail-closed). The dropped `block_if.any_critical_check_failed: true` encoded the pre-D3 coarse rule "any critical failure ⇒ BLOCK," which is false against the actual routing: `structural_arithmetic` and `currency_match` are critical yet ESCALATE on failure; only `check_amount_matches_total` (no adjustment) BLOCKs. Asserting that flag would enforce a lie and pressure the code toward it. `critical_checks`, by contrast, is *true* and worth asserting: it turns silent spec/code drift (a fourth critical check added in code but not the config, or vice versa) into a loud `PolicyError`. Float parsing of `score_below: 0.80` would reintroduce `0.1+0.2` error into the one comparison that decides escalation, so thresholds are Decimal like everything else.
**Boundary.** `amount_greater_than` is currency-naive (numeric compare against `action.amount.value`); safe because over-escalation is the fail-closed direction and cross-currency escalates on `currency_match` regardless. `score_below` is inert on structured-only inputs (D27 boundary); it is a live knob only on the served `raw_text` path. `retry.max_attempts` is the single live retry knob (Slice 6 threads it into the loop, replacing the hardcoded default).

**Principle — inert config is worse than none.** A config key that reads like a control but controls nothing is a latent lie: a reader trusts it, sets it, and nothing happens. The three policy keys resolve one rule three ways. `critical_checks` is **kept** because it is *asserted* — a live spec/code drift tripwire (`PolicyError` on mismatch). `block_if.any_critical_check_failed` is **rejected** because it was *false* — it encoded the pre-D3 "any critical ⇒ BLOCK" rule the actual routing contradicts. `on_cap_exhausted` is **dropped** because it was *inert* — `retry.py` hardcodes cap-exhaustion to human-routing and reads no such key, and its only alternative value (a dead-end BLOCK) was ruled wrong in the Slice 3 grill. Assert it, reject it, or drop it — never ship a knob that does nothing.

## D29 — LineItem.quantity is Decimal, not int
**Decision.** `LineItem.quantity` is parsed as `Decimal` from a string or int (no-float validator, like `TaxLine.rate`), not `int`.
**Reasoning.** Real invoices bill fractional quantities (2.5 consulting hours, 1.5 kg). As an `int`, such an invoice fails Pydantic construction and fail-closes to ESCALATE at the API boundary — a false escalation on a perfectly legitimate document. `quantity` is consumed by no deterministic check (`structural_arithmetic` sums `line_item.amount`, never `quantity × unit_price`), so widening it cannot change any arithmetic result — it only removes a false-escalation class. Float stays banned for the same reason as all money-adjacent values (D1).
**Boundary.** Quantity remains unverified against `unit_price × quantity == amount` in v1 (that cross-check is not performed); widening the type does not add a check, it only stops rejecting valid inputs.

## D30 — The agent is a thin LangGraph over a value-only, pluggable retry loop
**Decision.** The Slice 6 demo agent is a small LangGraph — `propose` (LLM proposes an action from a structured Invoice) → `gate` (runs the loop) → conditional edge → END on `allowed`, else a `human` node that `interrupt()`s for approve/reject. The retry loop is **`run_with_retry` generalized once**: it gains a `propose_value(invoice, action, reason) -> Money | str` callable whose default returns `reason.expected` (byte-identical to the Slice 3 deterministic fix), while the agent injects an LLM re-proposer. The fixer is **value-only** — it returns just the value for the field `reason.field_to_change` names, and `run_with_retry` owns the `model_copy` and the field validation. HITL is **approve/reject only**; approval is a graph-level resolution, never a mutation of the gate's Decision (D25). The checkpointer is in-memory (`MemorySaver`). A malformed LLM proposal or re-proposal fails closed to a human (D11).
**Reasoning.** Slice 3 already built the loop, cap, precedence, and the `agent_fixable`-vs-`source_invalid` short-circuit — so the agent must *reuse* it, not re-implement it in graph edges where it would drift. Making the fixer pluggable turns "the agent retries" from a fiction (the deterministic fixer retried) into the truth (the LLM re-proposes against the compiler-grade reason) with one source of loop truth. Value-only is the load-bearing constraint, not a style choice: `check_amount_matches_total` BLOCKs on `amount != total` **only when `adjustments` is empty** and ESCALATES otherwise, so a whole-`ProposedAction` fixer lets the LLM "answer" a block by *declaring an adjustment* — flipping BLOCK→ESCALATE and burning a human on exactly the slip the agent could self-correct (D3 inverted). And `action_type`/`invoice_number` are read by no check, so a whole-action fixer could drift them unseen. Returning only the value for the gate-named field forecloses both by construction. LangGraph's genuine contribution here is the `interrupt` (a real pause/resume for the human), not the loop; keeping the graph thin keeps that visible. The router stays the only mock, so CI is deterministic and the loop/decision logic runs for real.
**Boundary.** The stubbed router exercises the graph mechanics (loop reaches ALLOW, cap escalates, ESCALATE pauses, malformed fails closed). **Correction (claims audit):** this entry originally claimed real-LLM re-proposal quality was covered by a `live`-marked test — no such test existed when that was written. A live test now exists but targets the *proposal* step, not re-proposal, and as of this correction it has **never been executed**, so neither path has ever been exercised against a real model; the D9 live-test requirement is unsatisfied until `pytest -m live` runs green once. The claim is corrected now and will be earned by that first green run, not by writing more unrun tests. Value-only means the agent can only correct the single field the gate flags — sufficient in v1 because the sole `agent_fixable` block is the amount misread; a future block targeting a different field names it in `field_to_change` and the same mechanism applies. The agent works from a structured Invoice (extraction is Slice 1/7); it performs no real side effect (no duplicate-store write). Recording an approval is deliberately **not** wired to ALLOW — ALLOW means "consistent with the evidence," not "paid"; it is a separate post-payment call (Slice 6.5(b) reverses the earlier "record after ALLOW" line, D23).

## D31 — Frame stage: verify the action IS the thing the checks verify (a prior gate)
**Decision.** Before the content checks (§7), a **frame stage** runs two critical checks — `action_type_supported` (`action.action_type == approve_payment`) and `invoice_number_match` (`action.invoice_number == invoice.invoice_number`, exact, on schema-normalized values). Either failure **ESCALATES** and short-circuits: the checks table carries the frame rows only, `score` is `null`, and no content check, policy threshold, or total-grounding gate runs. Both frame reasons are `block_type=None`, `field_to_change=None`, `expected=None` — **never** `agent_fixable`. Identifier normalization (strip surrounding whitespace, `min_length: 1`) lives in a **schema field validator** on both `Invoice.invoice_number` and `ProposedAction.invoice_number`, not in the check. The short-circuit lives in `decide()`, not in the stage functions: `run_frame_checks` and `run_checks` each always return their full row set, so `critical_check_names()` (their union on a valid sample) cannot be truncated.
**Reasoning.** Every content check is written to verify one thing — an `approve_payment` action against the invoice it names — but nothing verified the action *was* that thing: `run_checks` reads neither `action_type` nor `action.invoice_number`, so `reject` with a matching amount returned ALLOW (the gate approving a rejection) and an action naming `INV-999` against source `INV-001` passed every check. These are **frame** failures ("is this even the thing the checks verify?"), categorically different from **content** failures ("is the verified thing right?"). They must **ESCALATE, not BLOCK**: a frame mismatch has two opposite fixes — the action is right and wrong evidence was attached, or the evidence is right and the action's field is a typo — and AgentGate cannot tell which, so it fails BLOCK's own D3 criterion (BLOCK is only for what the agent can actually correct). BLOCK would be actively dangerous: a value-only fixer (D30) with `expected="approve_payment"` would rewrite `action_type` and resubmit, converting a reject into an approve; with `expected=invoice.invoice_number` it would align the action to whatever invoice was attached, auto-healing a mismatch into an approval against an invoice nobody looked at. So `expected` is `None` (unfixable) and the human-facing comparison lives in the `message`. The frame runs **before** content and suppresses it because a content BLOCK ("set your amount to this total") fired against the wrong invoice or a rejection would point the agent at evidence nobody validated — so BLOCK > ESCALATE > ALLOW is a *within-content-stage* rule with the frame as a prior gate (a §7 amendment, not just a decision — D23). Normalization lives in the **schema** because `duplicate_store` keys on `invoice.invoice_number` as a SQLite TEXT primary key (BINARY collation): trimming only inside the check would leave `" INV-001"` and `"INV-001"` as distinct keys, and one leading space would silently defeat the very duplicate check the store exists for. Normalizing at the boundary fixes the store, `evidence_used`, and the check at once, and leaves the check a pure exact-match with no normalization logic — D8 applied to identifiers (surrounding whitespace is a lossless transport artifact; case/internal-spacing/punctuation stay significant, no entity canonicalization). Keeping the short-circuit in `decide()` is what stops `critical_check_names()` from silently dropping the content checks the day it introspects a sample that failed the frame.
**Boundary.** The frame checks are latent-safe until Slice 7a: the Slice 6 agent builds its action from the invoice (always `approve_payment`, matching `invoice_number`), so they never fire there — correctly, since an agent cannot self-fix a frame error; they become load-bearing only when a caller sets those fields freely over HTTP. `invoice_number_match` proves the right identifier is named, **not** that the attached invoice is genuine (that is the D5 independent-fetch milestone). `flag`/`reject` escalate uniformly rather than per-type — v1 verifies approvals only; a well-formed `reject` is *unsupported*, not *malformed*, so it returns a proper `escalate` Decision with an explanatory row, never a validation error.

## D32 — Decision.score is Optional; null = not computed
**Decision.** `Decision.score` is `Optional[Decimal]`. It is `None` (JSON `null`) exactly when the decision scored nothing — a frame-stage escalate (content checks never ran, D31) or a malformed-input fail-closed escalate (Slice 6.5(c) factory). A present `Decimal` (including `0`) means the `soft_pass_ratio × coverage` formula (D2/D16) actually ran.
**Reasoning.** A required score forces a number onto decisions that measured nothing, and the natural default `0.0` lies in the opposite direction from the truth: `0` reads as "we measured, everything failed," when the fact is "nothing was measured." That is the same category error as *unsupported* vs *malformed* (D31), dressed as a value. `null` is the only honest encoding and keeps `score` meaning one thing everywhere it is logged, traced, or displayed (the one-value-one-meaning discipline of D25). Landing it in Slice 6.5(a), not (c), avoids writing a test that asserts `score == 0` (or `1.00`) on a frame-escalate — a value already known wrong — only to delete it two commits later, which would be a D23 test-expectation edit against a spec line about to change anyway.
**Boundary.** Consumers must treat `null` as "not computed," never coerce it to `0`; the dashboard and any threshold logic branch on presence. `score_below` is unaffected — it evaluates only on the content path, where a real coverage number always exists.

## D33 — Duplicate store: collisions raise; one locked connection
**Decision.** `DuplicateStore.mark_approved` uses a plain `INSERT` (not `INSERT OR IGNORE`) and raises a typed **`AlreadyApprovedError`** when the invoice number is already recorded. The store keeps its **single shared connection**, opened with `check_same_thread=False`, and serializes every operation (`is_approved`, `mark_approved`, `close`) behind one `threading.Lock`. The store does **not** re-normalize keys — `invoice_number` arrives already schema-normalized (D31); one boundary owns normalization.
**Reasoning.** `INSERT OR IGNORE` silently swallowed the exact collision the store exists to detect — a duplicate-payment tracker that eats duplicate writes is the bug wearing the fix's clothes. Between raising and returning a bool, raising wins because a returned `False` is exactly as ignorable as `OR IGNORE` was — the failure mode just moves one level up the call stack; an exception cannot be dropped by accident. And v1 has no legitimate replay path (no queue, no auto-retry on the recording call), so a second write for the same key is never benign: it is either a caller bug or a double-payment that already slipped past the gate — both deserve a loud stop, not a shrug (fail-closed, D11's spirit at the write boundary). On threading: FastAPI runs sync endpoints in a threadpool, so the Slice-2 connection (default `check_same_thread=True`) raises `ProgrammingError` the moment a second thread touches it. The obvious fix — a fresh connection per operation — silently breaks the `:memory:` default (every new `:memory:` connection is a fresh empty database, so all tests and any in-memory use would lose state between calls). Keeping one connection with `check_same_thread=False` plus an external lock gives uniform behavior for `:memory:` and file paths, keeps every existing test's semantics, and is trivially correct at v1 volume — sqlite3 is safe across threads when access is externally serialized.
**Boundary.** The lock serializes all store access — fine for a single-process v1 service, not a multi-process/multi-writer design (that is the post-v1 Postgres/Supabase migration, PRD §4). `AlreadyApprovedError` is the *write-time* backstop, not the primary defense: the duplicate check at decision time (`is_approved` → ESCALATE) remains the gate's mechanism; the raise catches what slips between a stale read and the write. If a legitimate replay path ever appears (a queue with at-least-once delivery), the caller must catch `AlreadyApprovedError` and treat it as idempotent success — a deliberate, visible act, not a silent default.

## D34 — Fail-closed factory + uniform input bounds (the data boundary)
**Decision.** A pure `fail_closed_decision(errors: Sequence[Exception | str]) -> Decision` factory lives in `decision.py`: `decision=escalate`, `score=None` (D32), `checks=[]`, `evidence_used=[]`, `proposed_action=None`, and one `BlockReason` per error with `check="fail_closed"` and `block_type`/`field_to_change`/`expected` all `None`. Messages are **bounded and never echo raw caller input**: `ValidationError`s format from field locations + pydantic messages only (`include_input=False`), capped at 5 details × 200 chars with a "+N more" tail; other exceptions/strings cap at 300 chars. An empty error list raises `ValueError` (a reason-less escalate would be an unexplained escalate — programmer error, fail loud). Alongside it, **every caller-supplied text field is bounded at the schema**: `invoice_number` 1–100 post-strip, `vendor` ≤ 200, `date` ≤ 50, `currency` 1–10 (stripped/non-empty on `Invoice` too, matching `Money`), `description` ≤ 500, `agent_rationale` ≤ 2000, `adjustments` ≤ 20 items each JSON-serializing to ≤ 500 chars, and numeric strings (`Money.value`, `quantity`, `TaxLine.rate`) ≤ 50 chars before Decimal parse — with the `TaxLine.rate` junk-string parse fixed to raise a `ValidationError` instead of escaping as a raw `InvalidOperation`.
**Reasoning.** The §9 promise — malformed input becomes a valid escalate Decision, never a 5xx — needs two things the code lacked. First, a single pure place that converts "could not verify" into a Decision, so the HTTP layer stays thin and the contract is testable without FastAPI. Its empty `checks`/`evidence_used` and `None` action are the honest encoding: a check row claims a check ran, and echoing an unvalidated action is precisely the unbounded-caller-text exposure this slice closes. `fail_closed` reasons are neither `agent_fixable` (nothing mechanical to fix) nor `source_invalid` (the source is unreadable, not proven inconsistent) — the same category discipline as unsupported-vs-malformed (D31). Second, bounds: measured concretely, a 50k-char junk `Money.value` lands **inside the pydantic error message itself** (our validators interpolate `{v!r}`), so `include_input=False` alone cannot keep caller text out of traces — truncation in the factory is load-bearing, and input caps bound what the validators can echo in every other path. The bound is uniform across fields because the exposure is uniform: everything on the wire models flows to the Decision echo, check details, Langfuse, or the dashboard. Numeric strings get a pre-parse cap because a million-digit Decimal is also a CPU/rendering DoS (`structural_arithmetic` does real arithmetic on it). `adjustments` stay shape-agnostic (cap count + per-item serialized size) because v1 reads only their non-emptiness (D13) and typing them now would speculate on the deferred verification milestone. And the `InvalidOperation` leak matters because it defeats the whole design: an exception type the boundary doesn't catch turns "fail closed" back into "crash."
**Boundary.** The caps are generous ceilings, not format validation (`date` accepts any ≤50-char string; currency codes are not ISO-checked). Bounds reject at the schema, so an over-limit field fail-closes to a human rather than being truncated-and-processed — over-rejection is the chosen direction. `raw_text` is a function parameter, not a schema field; bounding what tracing records of it is a Slice 7a concern. The factory formats what it is given — a caller that hand-builds an unbounded plain-string error bypasses the ValidationError path's caps only up to the 300-char string cap.

## D35 — POST /verify always answers 200 with a Decision
**Decision.** The endpoint reads the raw request body itself (capped, D36), decodes it with `json.loads(..., parse_float=Decimal)`, validates the request models, and returns **HTTP 200 with a `Decision`** for every submitted body — verified or fail-closed. Framework body parsing and its automatic 422 are bypassed; `ValidationError`, JSON decode errors, and a final catch-all `Exception` handler all convert through `fail_closed_decision` (D34). Wrong path / wrong method stay plain 404/405. Every `Decimal` on the wire serializes as a JSON **string** (money `"1240.00"`, score `"0.97"` — pydantic v2's JSON-mode default, kept deliberately). `trace_id` (uuid4), `latency_ms`, and `timestamp` (UTC ISO-8601) are stamped at the boundary on every response, fail-closed included.
**Reasoning.** Three forces point at one shape. (1) The §9 promise is that the response to any verification attempt IS a Decision; FastAPI's own 422 would break that before our code ran, so the endpoint must own parsing. (2) Owning the parse is also what D1 requires: framework JSON turns the number `1240.00` into `float 1240.0`, which the `Money` validator rightly rejects — valid bodies would fail-close on a transport artifact; `parse_float=Decimal` hands the validator the literal text losslessly, so callers may send money as JSON numbers or strings. (3) One response shape keeps machine callers honest: an agent branches on `decision` and nothing else; if validation failures arrived as 422s without a Decision body, a naive caller would treat them as retryable transport errors and resubmit unchanged. The catch-all is the same promise held against ourselves: an exception class nobody anticipated must degrade to "cannot verify → escalate," not to a 5xx — the `InvalidOperation` leak (D34) already demonstrated how one uncaught type defeats the whole fail-closed design. Strings for Decimals because a JSON number is re-floated by the first `JSON.parse` that touches it — string-in (D1) implies string-out.
**Boundary.** 200-always applies to `POST /verify`, where a body was submitted for verification; transport-level misuse of other routes/methods is not a verification attempt and keeps ordinary HTTP semantics. The catch-all logs the exception server-side — fail-closed is not fail-silent; a swallowed-and-unlogged bug would just be a quieter crash. `GET /health` is a dumb liveness ping.

## D36 — The wire rejects what nothing reads
**Decision.** `extra="forbid"` on every caller-facing wire model — `Money`, `LineItem`, `TaxLine`, `Invoice`, `ProposedAction`, and the request envelope (`Source`, `VerifyRequest`). `source.po` is not accepted in v1. `source.raw_text` is bounded (≤ 50,000 chars) and treated literally when present — an empty string is evidence that grounds nothing (the D27 total gate escalates), never coerced to "absent." `line_items` ≤ 500, `tax_lines` ≤ 50. The request body is capped at 1 MiB, rejected on Content-Length when declared and re-enforced while streaming.
**Reasoning.** Silently dropping unknown fields is not liberal tolerance; at a trust boundary it is data loss with a concrete flip: a caller declaring a withholding who misspells `adjustments` gets the default `[]`, so `check_amount_matches_total` routes agent-fixable BLOCK instead of declared-adjustment ESCALATE, the value-only fixer (D30) "corrects" the amount to the full total, and the gate has laundered the exact overpayment the caller tried to declare. Rejecting `po` is the inert-config principle (D28) applied to evidence: accepting a document no check reads makes ALLOW claim more than was verified — "consistent with the evidence provided" must cover all of it. The `""`→absent coercion is rejected because it converts an escalate into a potential allow (the fail-open direction); an empty `raw_text` more plausibly means an upstream bug emptied real evidence than that the caller meant "none" — omitting the field is how "none" is said. The list caps and the body cap bound the Decision-echo/trace/dashboard exposure exactly the way D34 bounded strings; the byte cap is sized so no schema-valid request approaches it (worst case ≈ 0.5 MiB against 1 MiB), meaning it can never reject a valid body, only a flood.
**Boundary.** `Invoice.subtotal` stays accepted although no check consumes it: it is a field of the invoice document itself (§6 contract), not a separate evidence artifact promising a nonexistent check — that line is what separates it from `po`. Response-side models (`Decision`, `Check`, `BlockReason`) are not forbid — AgentGate constructs them. When `po_match` ships, `source.po` flips from rejected to read in the same commit.

## D37 — Tracing is an observer, never a gate
**Decision.** `tracing.py` exposes a tiny seam: a `Tracer` protocol with `NoopTracer` and `LangfuseTracer` (lazy import, optional `tracing` extra, pinned to the langfuse v2 API), `build_tracer()` (Noop unless both `LANGFUSE_PUBLIC_KEY` and `LANGFUSE_SECRET_KEY` are set and the client constructs), and `record_safely()`, which wraps every record call and swallows-and-logs every exception. The API traces the validated invoice/action dumps plus the `raw_text` **length only** as input, and the Decision as output; the fail-closed path traces body length only. `trace_id` is minted (uuid4) whether or not tracing is enabled, and doubles as the Langfuse trace id when it is.
**Reasoning.** The decision path must have zero externalities. Langfuse is a network service that can be down, slow, or misconfigured, and none of those states may change, delay, or crash a verdict — fail-closed governs *evidence*, not *observability*, and the failure directions deliberately differ: evidence failures escalate, tracing failures are logged and ignored (an untraced correct decision beats a traced non-decision). Key-gating to a no-op keeps CI and keyless self-hosts deterministic, free, and dependency-light (the same lazy-import pattern as litellm, D9). `raw_text` content is excluded because it is the one large caller blob, and its verification value is already fully captured in the grounding rows, the coverage score, and `evidence_used` — content in a third-party trace is pure exposure with zero added decision information; the fail-closed path records no body content for the same reason the factory echoes none (D34).
**Boundary.** Real Langfuse ingestion is verified manually against a live project, never in CI — the same line live LLM tests draw (D9); the tests exercise the seam instead (a raising tracer must not affect the decision; no `raw_text` content may appear in what the tracer receives). Export is batched with a flush on shutdown; a hard crash may lose traces, never decisions. Pinned to the v2 client API; revisit when the dashboard integration (7b) wants v3/OTEL.

## D38 — The HTTP surface reads the duplicate store; nothing over HTTP writes it
**Decision.** `create_app(store, tracer, policy)` is an app factory (tests inject; the module-level `app` for uvicorn wires from the environment once at startup). The store defaults to `:memory:`, with `AGENTGATE_DB_PATH` opting into a file. `/verify` calls `store.is_approved(invoice.invoice_number)` at decision time and feeds `decide(is_duplicate=...)`. There is **no** recording/mark-approved endpoint in v1.
**Reasoning.** The read is the duplicate check's live input and is honest at any store state — an empty store truthfully means "no known prior approvals." The write stays off the HTTP surface because v1 has no auth (§2): an unauthenticated recording endpoint would let anyone mark arbitrary invoice numbers as approved, poisoning the store so every future legitimate payment force-escalates — a denial-of-approval vector — and because recording is a post-payment fact owned by whatever executes the payment (the 6.5(b) reversal, D33): `/verify` saying "consistent" is not "paid." Defaulting to `:memory:` rather than a file keeps the server free of a silent side effect (a database file materializing in whatever directory uvicorn started from); with no HTTP writer the file would stay empty anyway, and an operator who wires a real post-payment recording process sets the env var deliberately.
**Boundary.** Per-worker `:memory:` stores are independent — irrelevant while nothing over HTTP writes — and a shared file via env is single-process-safe under the D33 lock; multi-writer remains the post-v1 Postgres migration. The duplicate check over HTTP is therefore inert until an operator populates the store out-of-band (documented, not hidden): the wiring is real and tested; the data arrives with the post-payment recording call.

## D39 — The dashboard is a dumb pipe that renders the wire truth
**Decision.** The dashboard submits **one textarea** holding the whole `VerifyRequest` JSON and sends its contents **verbatim** as the raw request body — no client-side parsing gate, no re-serialization, no pre-validation that blocks submission. "Upload" is a file input that loads a `.json` into that box; sample buttons load presets that mirror eval cases. Rendering rules: `score: null` displays as **"not computed"** (never 0); money values render as the exact wire strings (no `parseFloat`, no locale reformatting); checks and reasons render exactly as returned; `trace_id` is always copyable text and becomes a link only when `NEXT_PUBLIC_TRACE_URL_TEMPLATE` is configured; a fetch/network failure renders an **error panel, never a synthesized decision**.
**Reasoning.** The gate is the validator — a UI that parses first would be a second, weaker validator in front of it: unparseable input couldn't even be submitted, hiding exactly the fail-closed behavior the demo exists to show (paste garbage → watch a valid escalate Decision come back). Verbatim submission also means the UI can never mask or "helpfully fix" caller input — the byte-for-byte body is what the gate judged, so what you see is what was verified. The rendering rules are D1/D32 carried to the pixel: `parseFloat("1240.00")` quietly reintroduces the float the whole stack was built to exclude, and a `null` score coerced to `0` (or blank) tells the D32 lie in the one place humans actually look. An invented "decision" on network failure would be the UI overruling the gate — the empty state must stay visibly distinct from a verdict. The trace link is D28's inert-affordance principle in the UI: a link that 404s against an unconfigured Langfuse reads like a feature and behaves like a bug.
**Boundary.** Two-box (invoice/action) form UX was considered and rejected for v1: composing the envelope client-side requires parsing the parts, recreating the pre-validation problem. The dashboard has no decision history (Langfuse is the history), no auth, and no retry loop — the compiler-error loop is the agent's (Slice 3/6); the dashboard is an inspection surface.

## D40 — Direct browser→API with explicit-origin CORS; unset means same-origin only
**Decision.** The Vercel frontend calls the Render backend **directly from the browser**. CORS is granted only to the exact origins in `AGENTGATE_CORS_ORIGINS` (comma-separated; scheme+host+port exact — `localhost` ≠ `127.0.0.1`), `allow_credentials` off, methods `POST`/`GET`, headers `Content-Type`. When the variable is unset the middleware is **not installed at all** — no CORS headers, same-origin only. `create_app(cors_origins=...)` injects for tests; the module app reads the env once.
**Reasoning.** The alternative — a Next.js rewrite proxying `/api/*` to Render — inserts a server-side hop with its own ~30s timeout in front of a free-tier backend whose cold start can exceed 50s: the demo would 504 precisely on a recruiter's first click, while a direct browser call just waits and succeeds. Direct also keeps the deployed URL visible in the UI (transparency) and one moving part fewer. Explicit origins because `*` would let any page on the internet script requests against a deployment; no-credentials keeps the grant to the narrowest CORS category. Absent-config-means-absent-middleware is the fail-closed default and the D28 principle again: a CORS layer that exists but allows nothing is a knob pretending to be off — better for it not to exist.
**Boundary.** CORS here is exposure hygiene, not auth — v1 has no auth (§2), and anyone can call the API directly with curl regardless; origins only govern what *browsers on other sites* may do. The env is set to the real Vercel origin as the last deploy step (D42); local e2e sets it to the test origin explicitly.

## D41 — The frontend gate runs in CI against the real stack
**Decision.** The Playwright suite boots the **actual** stack via `webServer`: uvicorn (`app.main:app` — later `agentgate.main:app`, D43 — in-memory store, CORS env set to the test origin; command overridable via `E2E_BACKEND_COMMAND` so CI and the local venv differ only in an env var) plus `next dev`, on fixed test ports. Tests drive the PRD gate literally — upload a sample `.json` ⇒ the correct decision renders — plus the tampered-block, reject-escalate ("not computed" score), and garbage-body fail-closed paths. CI gets a second job: install backend `[server]` extras + `npm ci` + `playwright install chromium` + `npx playwright test`.
**Reasoning.** D24's rule is full-suite regression on every commit, and D22's is that a gate nobody runs is decorative — a frontend e2e that lives only on a laptop is both. Running against the real backend (not a mocked fetch) keeps the LLM-era discipline of this repo: the only legitimate mock is the router (D9), and this path contains no router — structured samples, deterministic decisions, in-memory store — so CI stays fast, free, and honest while covering the exact seam (CORS, verbatim body, wire serialization) unit tests can't.
**Boundary.** Chromium only (the gate is contract coverage, not browser-matrix QA). `next dev` rather than a production build keeps the job fast; production behavior is covered by the Vercel deploy itself (D42). Visual styling is untested by design — assertions bind to `data-testid` and rendered text, not pixels.

## D42 — Deploys are code: one Blueprint, one env-wiring order
**Decision.** Backend deploy is declared in `render.yaml` (free-plan web service rooted at `backend/`, `pip install -e ".[server,tracing]"`, `uvicorn app.main:app --host 0.0.0.0 --port $PORT` — later `agentgate.main:app`, D43; `AGENTGATE_CORS_ORIGINS` and `LANGFUSE_*` declared `sync: false` so values live only in the dashboard). Frontend deploy is a Vercel project on `frontend/` with `NEXT_PUBLIC_AGENTGATE_API` (+ optional `NEXT_PUBLIC_TRACE_URL_TEMPLATE`). Wiring order: (1) Render up → backend URL; (2) Vercel up with that URL baked; (3) set the Vercel origin into `AGENTGATE_CORS_ORIGINS` and restart. Documented realities: free-tier cold starts (~1 min first hit) and ephemeral disk (the duplicate store resets on redeploy — inert anyway, D38). The first live Langfuse-ingestion check happens at this deploy, manually (D37's line: live externals are never CI's job).
**Reasoning.** A deploy that exists only as clicks in two dashboards is unreproducible and unreviewable; the Blueprint pins the build/start commands and env *names* in-repo (values stay out — secrets never enter git). The wiring order isn't bureaucracy — it's the dependency graph: the frontend needs the backend URL at build time, and CORS needs the frontend origin, so any other order ships a broken hop. Cold starts and disk resets are stated up front because a demo that stalls 60 seconds unexplained reads as broken — the README says what "waking up" means.
**Boundary.** Deploy execution still requires account auth (Vercel login, Render Blueprint connect) — one-time human steps; the repo carries everything else. `PYTHON_VERSION` is pinned in the Blueprint; bumping it is a deliberate edit, not dashboard drift.

## D43 — The import package is `agentgate`, renamed atomically with its runtime references
**Decision.** The top-level Python package `app` is renamed to `agentgate` — code, tests, eval, `pyproject` package discovery, and **every runtime reference in the same commit**: the `render.yaml` startCommand, CI's `E2E_BACKEND_COMMAND`, the Playwright backend command, and the README instructions. The live deploy's health and a live decision are re-verified immediately after the push.
**Reasoning.** A pip-installable library that claims the site-packages name `app` collides with half the tutorial code on the internet — the audit flagged it (item (e)) the day packaging became a goal, and the fix belongs before the first `pip install` anyone else runs, not after. Atomicity is not tidiness, it is uptime: the Render Blueprint auto-syncs `render.yaml` on push and redeploys, so a rename split across commits has a window where the deployed start command imports a package that no longer exists — the rename would take down the live demo it is supposed to polish. One commit means the redeploy that picks up the new start command also carries the renamed package.
**Boundary.** The PyPI *distribution* name was already `agentgate`; this aligns the *import* name with it. Local editable installs must be refreshed (`pip install -e`) after the rename; the old `app.egg-info` is inert and ignored. Observed at deploy time: a single commit is atomic in git but **not** in Render, which applies the code push (auto-deploy) and the Blueprint sync (the new start command) as two separate operations — the rename's first deploy attempt ran the old start command against the renamed code and exited 1, then the Blueprint sync's own deploy went live two minutes later. Harmless in practice (a failed deploy never replaces the serving container), but the honest statement is: the atomic commit prevents a *broken live state*, not a *failed deploy attempt* in the window between the two applies.

## D44 — MCP wraps `decide()` in-process, under the same envelope and fail-closed contract as HTTP
**Decision.** The `verify_action` MCP tool takes `proposed_action` + `source`, validates them through the **same** `VerifyRequest` model as `POST /verify` (bounds, `extra="forbid"`, po rejected), calls `decide()` in-process, stamps the same boundary fields, and **always returns a Decision dict — it never raises to the MCP client**; every failure converts through `fail_closed_decision`. The duplicate store and tracer wire from the same env vars as the HTTP app (keyless tracing = no-op). Money over MCP must be JSON **strings**: the transport parses JSON before AgentGate sees it, so a numeric amount arrives as an already-materialized float and is rejected by the D1 validator into a fail-closed escalate whose message says to use strings. Packaging: optional `mcp` extra, console script `agentgate-mcp` (stdio), and a clean-venv install smoke job in CI.
**Reasoning.** Wrapping HTTP instead would chain the tool's availability to a running uvicorn and add a network hop that can fail in ways the gate cannot classify; `decide()` is pure and the MCP server is a *boundary*, so it gets exactly the boundary treatment the HTTP layer got (D35): validate, decide, stamp, fail closed — one contract, two transports. Never-raise matters because an MCP exception surfaces to the calling agent as a tool *error*, outside the Decision vocabulary: an agent seeing an error might retry or route around the gate, whereas a fail-closed escalate Decision keeps the verdict machine-readable and in-band. On floats: HTTP could parse the raw body with `parse_float=Decimal` (D35) because it owned the bytes; MCP hands us post-parsed dicts, and "helpfully" converting float→Decimal via `str()` would launder a value that already lost its literal text through the exact type D1 bans — rejection with an instructive message is the only D1-honest behavior, and it is the §4 lowest-common-denominator contract applied to a second transport.
**Boundary.** The store is per-process (same as the HTTP app, D38); an MCP server spawned per-client sees its own store unless `AGENTGATE_DB_PATH` points at a shared file. The tool is read-only like `/verify`. Tests exercise the tool function and its registration; full MCP wire-protocol conformance rides on the official SDK, not on our tests.

## D45 — Fetch mode: the agent chooses *which* invoice, the system of record says *what it contains*
**Decision.** Independent source fetch ships as a second mode of the **same** request envelope, not a new endpoint: `Source` becomes a validator-enforced union — caller mode (`invoice` present, `raw_text` optional) XOR fetch mode (`fetch: "<invoice-number>"` present and nothing else). In fetch mode the boundary resolves the record from the operator-configured system of record (`resolve_source()`, shared by HTTP and MCP) and passes the fetched invoice + the record's own `raw_text` to the unchanged, pure `decide()`. `fetch` combined with `invoice` **or** `raw_text` is a validation error. Every fetch failure — unconfigured, not found, missing directory, corrupt/oversized/ambiguous record — raises a typed `SourceOfRecordError` that the boundary converts through `fail_closed_decision`; not-found is **never** an agent-fixable BLOCK. Fetched decisions carry provenance on the wire: every `evidence_used` entry is prefixed `system_of_record:`, stamped at the boundary alongside `trace_id`.
**Reasoning.** This is the §0 trust-anchor upgrade: v1's honest gap is that the caller supplies the evidence it is judged against, so a doctored source plus a matching action passes every check. Fetch mode removes the caller from the evidence path — the agent names an identifier and controls nothing else. One envelope rather than a second endpoint because the fail-closed response contract (D35) is the hard-won invariant, and a second endpoint would be a second copy of it to keep honest; the mode is a property of the *source*, which is exactly what `source` describes. Rejecting mixed evidence is what makes the provenance prefix truthful — if caller `raw_text` could ride along, "system_of_record" would label evidence the caller shaped. The prefix lives at the boundary because `decide()` genuinely does not know where its invoice came from; provenance is a boundary fact like latency. Not-found stays a plain escalate because a fixer that retries identifiers until one resolves is D31's "edit the frame until it fits" trap wearing a new hat. `decide()` stays I/O-free so the core remains pure, deterministic, and CI-testable — the fetcher is injected at the boundary exactly like the store and tracer (D38).
**Boundary.** The trust anchor is exactly as strong as the deployment: it holds when the records store is writable only by the operator. The reference store is a local directory (D46), not a live ERP — a real connector is a later implementation of the same protocol. Caller-supplied mode remains, and remains consistency-checking only (§0's original claim, unchanged). Eval is untouched: the harness exercises `decide()`, which did not change.

## D46 — The reference system of record is a directory of schema-validated records, keyed by content, never by filename
**Decision.** `DirectorySourceOfRecord(path)` serves fetches from a directory of `*.json` files (`AGENTGATE_RECORDS_DIR`; unset = no store = fetch requests escalate). Each file is a caller-mode `Source` document (`{"invoice": …, "raw_text": …}`) parsed with `parse_float=Decimal` and validated through the **same schema and bounds as caller input**, `extra="forbid"` included; each file is capped at 1 MiB. The lookup key is the `invoice_number` **inside** each record — filenames key nothing. The directory is re-scanned on every fetch. Any unparseable, schema-invalid, or oversized file fails the fetch; so do two valid records claiming the requested number.
**Reasoning.** Content keying kills two birds: no path is ever constructed from caller input, so path traversal is structurally impossible (an identifier like `../secrets` is just a key that matches nothing), and the `invoice_number_match` frame check stays a live tripwire for store misfiles instead of becoming a tautology. Operator data gets caller-grade validation because trust in fetch mode *transfers* to the store — a malformed record must escalate, not crash (D11), and `extra="forbid"` is load-bearing here too: an operator's misspelled `raw_text` key silently dropped would yield grounding coverage 1.0, the fail-open direction. `parse_float=Decimal` is D1 applied to disk — a stored numeric `1240.00` must stay the exact Decimal of its literal text, same as over HTTP (D35). Failing the fetch on *any* invalid file is the fail-closed direction: the broken file might be the requested record, and a system of record with corrupt entries cannot anchor trust; silently skipping it would hide operator errors. Per-fetch scanning keeps operator edits live without a restart and is honest about scale — v1 stores are demo-sized, and the O(n) scan is replaced by a real connector, not optimized.
**Boundary.** Duplicate detection applies to the requested key (an ambiguity that affects the answer); a parseable record with a *different* duplicated key does not fail an unrelated fetch. No watch/caching layer, no write API — the operator edits files. The 1 MiB per-file cap mirrors the HTTP body cap: no schema-valid record approaches it.

## D47 — Inert spec is the same defect as inert config
**Decision.** The D28 rule — **assert it, reject it, or drop it** — applies to spec and documentation lines, not just config keys. A line asserting a component, tool, or behavior with no code behind it and no owning milestone must be made true, moved to an explicit milestone list, or deleted. Adopted after a full claims audit (every factual claim in the public docs grep-verified against code, live claims verified against the live deployment): a "locked" stack section had accumulated three inert assertions — an eval framework never installed or referenced anywhere (DeepEval), "fallback" providers with no fallback chain behind them (providers swap via one env var; nothing falls back), and a database/retrieval stack (Postgres/pgvector/Supabase) with zero code — and the "only mocked seam" claim had drifted from literal truth (D9, now reworded as substitution vs fault injection).
**Reasoning.** An inert config key lies to an operator who sets it; an inert spec line lies to every future maintainer — and to the spec-is-arbiter discipline itself: D23 resolves red tests against the spec, and an arbiter containing fiction corrupts the ruling. These claims accumulated precisely because nothing re-reads a spec section against reality once it stops being edited; gate-driven development verifies new behavior, never old claims, so claims rot exactly where enforcement doesn't reach (the same mechanism that let a configured-but-empty `live` test marker stand for nine slices). The audit that caught all of this — claim by claim, grep-and-confirm, with live curls for live claims — is cheap enough to repeat and found in one pass what nine slices of green gates did not.
**Boundary.** This entry records the standard a claim must meet to stay in the docs; it does not schedule recurring audits. Market and positioning statements ("mirrors real startups in the space") are opinions, not auditable facts, and are out of scope for the rule.

## D48 — The demo agent ships as a runnable module, not a README snippet
**Decision.** `agentgate/agent/demo.py` (`python -m agentgate.agent.demo`) drives the real LangGraph agent and the real gate end to end with a live model behind the router, through two scenarios: a clean invoice (the model reads it and proposes; the gate verifies — a misread comes back as a machine-fixable block that the retry loop corrects and resubmits), then an invoice over the policy amount ceiling (however correct the proposal, the gate escalates per policy and the graph pauses for a stdin approve/reject). The module narrates whatever actually happened; it never scripts an outcome. Seams are injectable (`llm_call`, `ask`, `out`) so tests drive it with canned messy model output through the router while the graph and gate run unmocked (D9). Missing optional extras exit with an instructive message; a missing or dead provider key is deliberately **not** preflighted — it surfaces as the typed router error through the graph's fail-closed human route, D11 demonstrating itself. Money and the score render as the exact wire strings; a null score prints "not computed" (D32/D39 apply at the terminal too).
**Reasoning.** The demo exists to be recorded (the README GIF) and reproduced: a committed module means the command a viewer sees typed is a command a cloner can run, whereas a pasted snippet in a recording documents code the repo does not contain. Narrating the actual outcome keeps a nondeterministic live model from ever making the recording lie — if the model misreads, the demo shows the block-and-fix loop, which is the product working, not the demo failing.
**Boundary.** The demo is repo surface, not product surface: no console script, no place in the served API, and it runs the graph as the graph is — structured invoices only, no duplicate store, no raw-text grounding (the dashboard demo covers those paths). The deterministic block-then-fix recording remains the dashboard GIF's job; this module's scripted-fixture twin lives in its tests.

## Roadmap re-sequence (risk-first, thesis-first)
**Decision.** Replace the naive bottom-up MVP order with 8 slices, each with one test gate: (1) thin end-to-end skeleton with a grounding-only result; (2) deterministic checks + block-with-reason; (3) crude retry-loop demo; (4) honest eval; (5) policy engine; (6) LangGraph agent + HITL; (7) dashboard + deploy; (8) MCP + packaging.
**Reasoning.** The credibility of the project lives in grounding and the honest eval, and the hero feature is the compiler-error retry loop — all of which the old order buried behind scaffolding and surface. The genuine risk is whether extraction + grounding can get a trustworthy number out of a messy real source; that must be de-risked on day one via a thin walking skeleton, not after two slices of plumbing. So the thesis-proving and risky work moves to the front; policy, agent, dashboard, and MCP move behind it. Each slice keeps exactly one isolated, falsifiable test gate, preserving the test-gate discipline.
**Boundary.** Slices 1–3 deliberately ship crude, disposable versions (grounding-only decision, scripted retry) ahead of their polished counterparts; the hardened agent/HITL, dashboard, and packaging come later by design. Three later refinements to this list, kept honest here: **Slice 6.5** was inserted to close the gate's contract before HTTP exposure (D31–D34 — frame stage, `score` optional, store hardening, fail-closed factory + input bounds), the old fat Slice 7 was **split into 7a** (thin read-only FastAPI) **and 7b** (dashboard + deploy) per that slice's own too-fat warning, and a **ninth slice — independent source fetch (D45/D46)** — followed after the v1 scope closed, shipping the trust-anchor upgrade the threat model names. PRD §10 is authoritative for the current order.
